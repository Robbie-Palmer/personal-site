# Summary

Use **[Neon](https://neon.com) Postgres** as the application data store, accessed via
**[Drizzle ORM](https://orm.drizzle.team)** (Postgres dialect) with
**[Hyperdrive](https://developers.cloudflare.com/hyperdrive/)** for connection pooling from
Cloudflare Workers.

Recipe content moves to the database. `.cook` files remain the canonical *format* (Cooklang
syntax preserved as a text column), but the database becomes the canonical *storage location*.
Existing `.cook` files serve as the import source for the initial recipe corpus.

# Context

The recipe site needs a relational store for:

* users, sessions, and OAuth accounts
  ([ADR 032](/projects/recipe-site/adrs/032-better-auth))
* household membership and sharing metadata
  ([ADR 035](/projects/recipe-site/adrs/035-authorization-model))
* recipe content — including user-created drafts and published recipes
* user-owned mutable state: favorites, cooking logs, portion preferences

The first authenticated feature is **recipe editing through the UI**. This requires recipe
content to live in a writable data store, not only in git-committed `.cook` files.

[ADR 030: Cooklang](/projects/recipe-site/adrs/030-cooklang) originally positioned `.cook` files
as the permanent source of truth. The recipe site PRD anticipated this evolution: *"Leverage tech
debt — start with recipes as version-controlled structured content. Move to a richer content
workflow when the constraint actually bites."* UI-based recipe authoring, particularly for
non-technical household members, is the moment the constraint bites.

# Decision

## Neon Postgres

Neon is serverless Postgres: real Postgres (not a Postgres-compatible wire protocol), with
scale-to-zero, database branching, and a generous free tier.

| Feature           | Value                                                 |
| ----------------- | ----------------------------------------------------- |
| **Engine**        | PostgreSQL 17                                         |
| **Free tier**     | 0.5 GB storage, 100 CU-hours/month, 2 CU max          |
| **Scale-to-zero** | Suspends after 5 minutes of inactivity                |
| **Branching**     | Instant database branches for PR preview environments |
| **Launch tier**   | $0.106/CU-hour + $0.35/GB-month (pay-as-you-go)       |

Neon was acquired by Databricks in May 2025 for \~$1B. The product continues independently and
prices dropped post-acquisition. The long-term direction trends toward Databricks' Lakebase
product, but standard Postgres means `pg_dump`/`pg_restore` to any other Postgres provider at
any time. Data portability does not depend on Neon's corporate trajectory. See
[ADR 037](/projects/recipe-site/adrs/037-backend-platform-for-authenticated-features) for full
acquisition context.

## Drizzle ORM with Postgres Dialect

Drizzle is used with the Postgres dialect from day one. Starting with Postgres avoids the schema
rewrite debt that would come from starting on SQLite/D1: Drizzle uses different table constructors
per dialect (`sqliteTable()` vs `pgTable()`), and the community consensus is that dual-dialect
maintenance is not worth the overhead.

## Hyperdrive for Connection Pooling

Hyperdrive maintains persistent connection pools from Cloudflare's global network to Neon,
eliminating per-request TLS handshake overhead. Included in the Workers Paid plan at no additional
cost. Neon's serverless driver (`@neondatabase/serverless`) is available as a direct-connection
fallback.

## Recipe Content in the Database

Recipes are stored in the database with Cooklang syntax preserved as a text column:

* The `cooklang-rs` WASM parser continues to parse recipe text client-side — no change to the
  parsing pipeline
* Structured metadata (title, cuisine, servings, tags, attribution, status) lives in typed columns
* Draft/published workflow is a status column, not a git branch
* Version history is a future feature (tracked via a revisions table or audit log, not git commits)

Existing `.cook` files serve as the import source for the initial recipe corpus. New recipes
created through the UI are born in the database. This is a revision of
[ADR 030's](/projects/recipe-site/adrs/030-cooklang) original stance — see the revision note on
that ADR.

# Alternatives Considered

## Cloudflare D1

D1 is Cloudflare's SQLite-based database. It was the default data store candidate if the runtime
stayed on Workers.

**Rejected because:**

* SQLite dialect, not Postgres. The Postgres ecosystem (full-text search extensions, vector search,
  DuckDB, PostGIS for other projects) is explicitly valued.
* Starting on D1 creates Drizzle schema migration debt: moving from `sqliteTable()` to `pgTable()`
  requires rewriting all schema files and regenerating all migrations.
* Better Auth has additional integration friction on D1 (internal metadata table conflicts,
  per-request instance creation, CLI tooling limitations).

D1 is a strong product for projects that want everything on Cloudflare with SQLite. It is the wrong
starting point when Postgres is the target.

## Integrated Backend Platform (Convex)

Evaluated and rejected in
[ADR 037](/projects/recipe-site/adrs/037-backend-platform-for-authenticated-features). Convex is a
document database (no SQL, no JOINs, no foreign keys) — a fundamental mismatch for relational auth
and sharing data.

## Recipes Stay in Git

Keeping `.cook` files as the permanent source of truth and having the app commit to git via the
GitHub API was considered. Rejected because:

* Non-technical household members cannot use a git-based authoring workflow
* GitHub API adds latency and complexity to every save operation
* Draft/published workflow maps naturally to a database status column, not git branches

# Consequences

## Positive

* **Full Postgres from day one.** FTS, vector search, and the broader Postgres ecosystem are
  available without migration.
* **Recipe editing unblocked.** Users can create and edit recipes through the UI. Drafts live in
  the database with no git dependency.
* **Cooklang compatibility preserved.** The parser reads text — it does not care whether that text
  comes from a file or a database column.
* **Schema-first with Drizzle.** Type-safe queries, generated migrations, and Drizzle Studio for
  data inspection.
* **Free at MVP.** Neon's free tier covers the expected usage for a long time.

## Negative

* **Second vendor.** Neon adds a platform relationship alongside Cloudflare. Mitigated by standard
  Postgres portability.
* **ADR 030 revision.** The Cooklang ADR's original "recipes stay in git" stance no longer holds
  for user-created content.
* **No git-native version history.** Recipe edit history moves from git commits to a
  database-backed mechanism (future feature). Until that is built, edit history is implicit in
  Drizzle migrations and application audit logic.

# When To Revisit

Revisit if any of the following become true:

* Neon's product direction changes materially post-Databricks acquisition
* the free tier is outgrown and the Launch tier pricing is not competitive
* D1 adds Postgres compatibility or Drizzle adds seamless dialect switching
* a concrete schema reveals limitations of the Neon/Postgres approach
