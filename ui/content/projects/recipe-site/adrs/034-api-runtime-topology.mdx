# Summary

The recipe site's authenticated backend runs on a **Cloudflare Worker** using
**[Hono](https://hono.dev)** as the HTTP framework. The static frontend stays on Cloudflare Pages.
Heavy processing lives on separate services.

# Context

The current architecture is intentionally simple:

* static Next.js export
* hosted on Cloudflare Pages
* no database-backed user state

That architecture should be preserved for the core browsing experience. What changes is the
introduction of a **thin dynamic layer** for:

* OAuth callbacks and session handling
* authorization checks and sharing mutations
* recipe CRUD (drafts, publishing, editing)
* authenticated per-user writes (favorites, cooking logs)

The key question was:

> Where should the privileged auth/API runtime live?

# Decision

## Deployment Shape

| Layer                  | Technology        | Role                                                                                    |
| ---------------------- | ----------------- | --------------------------------------------------------------------------------------- |
| **Static frontend**    | Cloudflare Pages  | Next.js static export, Cooklang WASM parser, PostHog proxy                              |
| **Auth/API backend**   | Cloudflare Worker | Hono, Better Auth, recipe CRUD, user state, authorization                               |
| **Connection pooling** | Hyperdrive        | Persistent pools from Workers to Neon Postgres                                          |
| **Database**           | Neon Postgres     | All application data ([ADR 033](/projects/recipe-site/adrs/033-application-data-store)) |

## Why Workers

Workers are the natural choice because:

1. **Platform continuity.** Pages, R2, DNS, Images, and Terraform are already on Cloudflare.
   Workers add the API layer without introducing a new hosting platform.
2. **Near-zero cold starts.** V8 isolates start in single-digit milliseconds. Auth flows are fast.
3. **Cheapest option.** Workers Paid ($5/month) covers the entire backend including Hyperdrive.
4. **The backend is thin.** With heavy processing on separate services, Workers only handles
   request/response auth and CRUD — well within its capabilities.

## Why Hono

Hono is a lightweight, TypeScript-native HTTP framework that runs on Workers, Node.js, Deno, and
Bun. This portability is the key property: if the runtime ever changes from Workers to Node.js
(via Cloudflare Containers or external hosting), the handler code migrates with minimal changes.

## Split Architecture

Heavy processing does **not** run on the Worker:

* **ML pipeline** (`ml-pipelines/recipe-parsing/`) stays Python/DVC
* **Future import jobs, background workers** use separate services when needed
* **GPU compute** uses dedicated infrastructure (Modal, RunPod, or CSP GPU instances)

This separation means Workers' runtime constraints (30s CPU time, V8 isolate, no native TCP) do
not limit the broader system architecture.

## Trust Boundary

The Worker is the server-side trust boundary:

* Auth flows, authorization checks, and privileged writes run in the Worker, not the browser
* The browser is a user interface, not the enforcement point
* Session cookies are `HttpOnly`, `Secure`, and server-owned
  ([ADR 036](/projects/recipe-site/adrs/036-application-security-baseline))

A direct client-to-database architecture was rejected. Even Neon Postgres with row-level security
would not provide the same level of control as explicit server-side authorization logic.

# Escape Paths

The architecture is designed for incremental migration:

**Workers → Cloudflare Containers.** If V8 isolate constraints compound, Containers (GA April 2026)
provide full Docker/Node.js on Cloudflare. Same Neon database, same Hyperdrive connection pooling,
Hono code largely portable. Trade-off: 1-3 second cold starts vs near-zero.

**Workers → External Node.js.** Hono runs on Node.js. Drizzle + Neon Postgres stays the same.
Redeploying the Hono app to any hosting (Fly.io, AWS, GCP) changes the runtime without changing the
application code or database.

**Cloudflare → Fully External.** Pages → Vercel/Netlify; Workers → any Node.js host; Neon stays.
Each layer migrates independently.

# Consequences

## Positive

* **Static-site benefits preserved.** Public recipe browsing stays fast, cacheable, and CDN-served.
* **Dynamic complexity stays isolated.** Only auth, CRUD, and sharing require the Worker. The
  majority of the site is still static.
* **One platform.** Frontend and backend are on Cloudflare — one account, one deployment pipeline.
* **Portable handler code.** Hono's multi-runtime support means the API layer is not locked to
  Workers.

## Negative

* **V8 isolate constraints.** Workers cannot run arbitrary Node.js code. The `nodejs_compat` flag
  and `better-auth-cloudflare` package address known issues, but friction may accumulate.
* **Local development complexity.** The pure-SSG local dev experience becomes more involved once
  Workers, Hyperdrive, and Neon are in the picture. Wrangler's local dev mode and Neon's branching
  help but add setup steps.
* **Two deployment artifacts.** The static frontend (Pages) and the API backend (Worker) deploy
  separately. This is intentional (the frontend should not redeploy when API code changes) but
  adds CI/CD surface.

# When To Revisit

Revisit if any of the following become true:

* V8 isolate constraints cause repeated friction that `better-auth-cloudflare` cannot address
* the backend grows beyond thin auth/CRUD into heavy server-side logic
* Cloudflare Containers mature enough to replace Workers as the primary runtime
* the app adds WebSockets, long-running connections, or streaming responses that Workers handle
  poorly
